# Introduction

The matcher matches products to potential investors. There are two outputs:

1. **Ranked potential investors** — Clients sorted by "likeness to buy" for a given product.
2. **Client suitability proposal** — How a product fulfills identified client needs, including scenario analysis.

First, we identify needs of a list of investors. Then we find the product(s) that best fit those needs.

We introduce a **4-dimensional framework** beyond traditional "risk vs. return" to provide a comprehensive **Product Suitability Map** for private clients:

| **Dimension** | **Key Definition** | **The Client's Core Question** |
|---|---|---|
| **Return** | Annualized expected return (CAGR) | "How much wealth will this build?" |
| **Risk** | Downside deviation (1y semi-deviation) | "How bad can the losses get?" |
| **Certainty** | Probability of achieving a target return over a given horizon | "How sure am I the money will be there?" |
| **Liquidity** | Speed/cost of converting back to cash | "Can I get out if I need to?" |

# Product Catalog Generation

## Overview

The product catalog is a CSV file generated by running:

```bash
.venv/bin/python -m src.main run-market-data --config config/config_marketdata.yaml
```

**Code path:**
- CLI entry: `src/main.py` → `run-market-data` subcommand
- Config model: `src/planbot/market_data_module.py` → `MarketDataConfig` (Pydantic)
- Main function: `get_market_data()` (line ~200)
- Config loader: `get_market_data_from_config()` (line ~393)

The catalog is a single CSV with one row per ticker, containing historical performance metrics, rating columns (`risk_rating`, `expected_return`, `liquidity_rating`), and optional certainty columns.

## Configuration: `config/config_marketdata.yaml`

### Ticker Selection

```yaml
execute_ticker_groupname:           # single string or list
  - selected_etf                    # merge tickers from all listed groups
  - demo-market-1Jun26

ticker_groups:
  selected_etf:
    - XLK, XLF, VOO, HYG, ...       # named ticker lists
  demo-market-1Jun26:
    - NEAR, JPST, NVDA, ...
```

- **`execute_ticker_groupname`** accepts a single group name (string) or a list of group names.
- When a list is given, tickers from all groups are **merged** (duplicates removed, order preserved).
- The output filename uses the **first** group name for the `<tickers_groupname>` placeholder.
- Can be overridden at CLI: `--tickers SPY QQQ` or `--ticker-groupname demo-market-1Jun26`.

### Data Fetching

```yaml
frequency: 1w           # yfinance interval: 1d, 1w, 1m, 1q
periods:                # historical windows to compute metrics for
  - 6m
  - 1y
  - 3y
  - 5y
  - 10y
metrics:                # computed per period
  - return              # period total return %
  - CAGR                # compound annual growth rate %
  - max_drawdown        # maximum peak-to-trough %
  - calmar_ratio        # CAGR / |max_drawdown|
  - downside_risk       # annualized downside semi-deviation %
  - volatility          # annualized volatility %
name_preference: long   # "long" or "short" for Yahoo Finance name field
asset_class_proxy:      # fallback ticker when history data is missing
  MONEYMARKET: SGOV
```

### How Ratings Are Assigned

All rating tables follow the same pattern: entries are evaluated top-to-bottom, and the **first matching row** wins. The last entry with a null threshold is the catch-all.

```yaml
# Example: downside deviation 15% → runs through until ≤20% → rating 4
risk_rating:
  - 1: 1%     # ≤1% → rating 1
  - 2: 4%     # ≤4% → rating 2
  - 3: 10%    # ≤10% → rating 3
  - 4: 20%    # ≤20% → rating 4
  - 5:        # >20% → rating 5 (catch-all)
```

### Risk Rating

**Source code:** `_estimate_risk_rating()` (line ~830)

Computed from **annualized downside deviation** (semi-deviation) from the 1y period:
- Falls back to the longest available period if 1y downside_risk is unavailable.
- Falls back to **max_drawdown** with the same config thresholds if no downside_risk data is available (too few data points).
- The config table (`risk_rating` in YAML) maps downside deviation % thresholds to 1-5 ratings.

**Additional boundary rule — SGOV floor** (`_enforce_sgov_return_ratio_rule`, line ~920):
- For ETFs only: if the 1y absolute return exceeds the SGOV (risk-free) return by a large margin, the risk rating is raised to at least `ceil(|1y_return / SGOV_return|)`. This prevents low risk ratings on high-return ETFs.

### Expected Return

**Source code:** `_estimate_expected_return()` (line ~881)

Raw **3Y CAGR** percentage (not a 1-5 score). Formatted to 2 decimal places.

### Certainty Rating

**Source code:** `_estimate_certainty_rating()` (line ~896)

#### Method: `zscore`

Uses the z-score formula:

$$z = \sqrt{T} \cdot \frac{r - \mu}{\sigma}$$

| Symbol | Meaning |
|---|---|
| $T$ | Horizon in years (e.g., 3y → 3.0) |
| $r$ | Fixed target return % from `certainty_target_return` (default: 0 = break-even) |
| $\mu$ | Horizon-matched historical CAGR |
| $\sigma$ | Horizon-matched annualized volatility |

- $\mu$ and $\sigma$ come from `_get_cagr_and_volatility_for_horizon()` (line ~886): exact period match preferred, otherwise closest period by month distance.
- The resulting z-score is compared against `certainty_rating` thresholds in the YAML.
- Code defaults (z-score thresholds): `-2, -1, 0.5, 1.5` — meaning higher thresholds → higher certainty rating.
- With $r = 0$, products with high CAGR get negative z (high certainty); low/negative CAGR gets positive z (low certainty).
- The $\sqrt{T}$ factor means certainty **grows with horizon** for positive-CAGR products.

**Config (defaults in code):**

```yaml
certainty_rating:
  - 5: -2     # z ≤ -2 → rating 5 (High Certainty, >97.5%)
  - 4: -1     # -2 < z ≤ -1 → rating 4 (Probable, 84-97.5%)
  - 3: 0.5    # -1 < z ≤ 0.5 → rating 3 (Neutral, 30-84%)
  - 2: 1.5    # 0.5 < z ≤ 1.5 → rating 2 (Speculative, 7-30%)
  - 1:        # z > 1.5 → rating 1 (Hope-Based, <7%)

certainty_target_return: 0   # r = 0% (break-even)
```

#### Certainty Boundary Caps

**Source code:** `_apply_certainty_cap()` (line ~944)

Certainty is capped at **3** for:
- Non-short-duration bonds (asset class contains "bond" or "government", not "short" or "moneymarket")
- Any product with risk_rating > 2

#### Configuration Switches

```yaml
certainty_enabled: false    # set true to include certainty columns in CSV
certainty_target_return: 0  # fixed target return %
certainty_period:           # which horizons to compute certainty for
  - 1y
  - 3y
  - 8y
```

When `certainty_enabled: false`, no `certainty_*_rating` columns appear in the output CSV.

### Liquidity Rating

**Source code:** `_estimate_liquidity_rating()` (line ~983)

**Primary:** Asset-type-based mapping from YAML. Matches case-insensitively against Yahoo Finance's `quoteType` and `assetClass` fields, then substring match as fallback. If no mapping matches, defaults to **3** (neutral).

## Output CSV Columns

The generated CSV contains these columns (ordered):

1. **Base columns:** `ticker`, `asset_class`, `name`, `currency`, `last_closing_price`
2. **Metric columns:** `{period}_{metric}` for each period × metric combination (e.g., `1y_return`, `3y_cagr`, `5y_max_drawdown`)
3. **Rating columns:** `risk_rating`, `expected_return`, `liquidity_rating`
4. **Certainty columns (if enabled):** `certainty_{period}_rating` for each period in `certainty_period`
5. **`last_update_date`**

Output path format: `runs/market_data/<tickers_groupname>.csv`

# Needs

For needs, liquidity 1 means it's highly unlikely we need the fund before the planned date. While 5 means the need may come earlier and needs liquidate the position to get the funding.

| Investment Objective | Typical Horizon (y) | Return (1-5) | Volatility (1-5) | Certainty (1-5) | Liquidity (1-5) | The Strategic Logic |
|---|---|---|---|---|---|---|
| University Fund / Mortgage down payment | 10–15 | 3 | 2 | 5 | 2 | Horizon-Locked: High certainty is achievable through balanced funds over 10 years. |
| Business Owner | 1–2 | 1 | 1 | 5 | 5 | Operating Buffer: Certainty must be near 100% (Contractual) due to the short horizon. |
| Retirement (Accumulation) | >7 | 5 | 5 | 5 | 1 | Growth Dominance: Can trade liquidity for returns; time creates the certainty. |
| Retirement (Distribution) | Ongoing | 3 | 2 | 4 | 3 | Cash Flow: Needs a high certainty of yield/coupons to fund monthly life. |
| Estate/Legacy | 30+ | 5 | 5 | 5 | 1 | Multi-Gen: Max volatility is ignored to harvest the highest long-term returns. |
| Emergency Fund | 0.5-1 | 1 | 1 | 5 | 5 | Immediate Need: No time for mean reversion; requires absolute price stability. |
| Tax Liability Reserve/Loan | <1 | 1 | 1 | 5 | 2 | Fixed-date obligation; matching duration is critical, not return. |

"Bridge to social security" is ignored for the moment as we need to account for the difference of each country.

# Product-need matching

The matching will be done by Weighted k-NN (NumPy/scikit-learn).

We need to put some weight on certain factors. To avoid over-engineer the parameter, we may just overweight certainty and liquidity as this is the major decision for many needs that has a hard horizon (retirement, university fund).

## Interpolation logic for matching

Given an investment objective with horizon **H years** (between 1 and 5), you can calculate a **Certainty score** for each asset class as:

```plaintext
Certainty(H) = Certainty-1y + (Certainty-5y - Certainty-1y) × (H - 1) / 4
```

### Example:

For **Public Equities** (C1=1, C5=4):

- At H=1 year → Certainty = 1.0
- At H=3 years → Certainty = 1 + (4-1) × (2/4) = 1 + 3 × 0.5 = **2.5**
- At H=5 years → Certainty = 4.0

Anything below 1y we take 1y-certainty. Similar we take 5y for any needs beyond 5y. This is a flat extrapolation.
